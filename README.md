## AI-Native DevOps Assignment 4 – Secure API Platform with Kong on Kubernetes

This repository contains a self-managed internal API platform built on Kubernetes using **FastAPI + SQLite** for the microservice and **Kong OSS** as the API gateway, with **JWT auth**, **IP-based rate limiting**, **IP whitelisting**, **custom Lua logic**, and **NGINX Ingress + ModSecurity** for DDoS protection. Everything is deployed declaratively via **Helm** and **Terraform**.

> Note: `ai-usage.md` must be written manually by you (the candidate) and **must not** be generated by AI.

---

## 1. High-Level Architecture Overview

- **Client**  
  - Any HTTP client (curl, Postman, browser) calling the platform entrypoint.

- **Ingress + DDoS Protection (NGINX + ModSecurity)**  
  - `k8s/ingress-nginx-modsecurity.yaml`.  
  - NGINX Ingress Controller fronting all external traffic.  
  - ModSecurity rules (in a `ConfigMap`) perform simple IP-based DDoS-style protection by counting requests per IP and returning `429 Too Many Requests` when thresholds are exceeded.

- **API Gateway – Kong OSS (DB-less)**  
  - Deployed via Helm chart in `helm/kong`.  
  - Configured with declarative `kong.yaml` (embedded via ConfigMap) in DB-less mode.  
  - Responsibilities:
    - Route traffic to the `user-service` microservice.
    - Enforce **JWT authentication** on protected endpoints.
    - Apply **IP-based rate limiting** and **IP whitelisting**.
    - Expose **public** (auth-bypassed) endpoints.
    - Run a **custom Lua plugin** for header injection and logging.

- **User Microservice (FastAPI + SQLite)**  
  - Source code in `microservice/app/main.py`.  
  - Containerized with `microservice/Dockerfile`.  
  - Uses a local file-based **SQLite** database (`sqlite.db`) for users and password hashes.  
  - Initializes DB and a default user on startup.

- **Infrastructure as Code**  
  - **Helm**:  
    - `helm/user-service` – deploys the FastAPI user microservice (`Deployment` + `Service`).  
    - `helm/kong` – deploys Kong with declarative config and Lua plugin.  
  - **Terraform** (`terraform/main.tf`):  
    - Creates namespaces (`platform`, `kong`, `ingress-nginx`).  
    - Applies a `NetworkPolicy` to only allow traffic to `user-service` from Kong and NGINX namespaces.

High-level request path:

`Client → NGINX Ingress + ModSecurity → Kong Gateway → User Service (FastAPI + SQLite)`

## 2. Microservice APIs

The **user service** exposes:

### 2.1 Authentication APIs

- **`POST /login`**  
  - Body: `application/x-www-form-urlencoded` (`username`, `password`).  
  - Validates user credentials against SQLite (`users` table with bcrypt hashes).  
  - Returns a **JWT**:
    - `sub`: username
    - `iss`: `JWT_ISSUER` env (default `user-service-client`)
    - `aud`: `JWT_AUDIENCE` env (default `internal-clients`)
    - Signed using `JWT_SECRET` (HS256).

- **`GET /verify`** (microservice-level)  
  - Query: `?token=<jwt>`.  
  - Validates the token signature and claims and returns `{"valid": true, "claims": ...}` on success.  
  - Configured as a **public** endpoint at Kong (no gateway-side JWT auth).

### 2.2 Protected User API

- **`GET /users`**  
  - Requires a valid **Bearer JWT**.  
  - Returns the list of users from SQLite (id + username).  
  - Kong’s JWT plugin enforces token validity before the request reaches this endpoint.

### 2.3 Public Health API

- **`GET /health`**  
  - Returns a simple JSON `{ "status": "ok" }`.  
  - Public, no auth enforcement at Kong.

## 3. API Request Flow (Client → Kong → Microservice)

1. **Client call**  
   - Client sends HTTP request to the platform external address (exposed via Kubernetes LoadBalancer or NGINX Ingress IP/hostname).

2. **NGINX Ingress + ModSecurity**  
   - For each incoming request, ModSecurity rules:
     - Increment an IP-scoped counter.
     - If the counter crosses a threshold (e.g., >100 in a short period), immediately return **429**.  
   - If allowed, NGINX forwards the request to **Kong’s Service** (Kubernetes Service defined by the Helm chart).

3. **Kong Gateway**  
   - Uses the declarative `kong.yaml` to:
     - Match the route (based on path and method).  
     - Apply relevant **plugins**:
       - **JWT plugin** for `/users` route.  
       - **Rate-limiting plugin** for `/users` route.  
       - **IP-restriction plugin** at the service level.  
       - **Custom Lua plugin** for all routes on the service.

4. **User Service**  
   - If authorized and not rate-limited/restricted, Kong proxies the request to `user-service` Kubernetes Service.  
   - FastAPI application handles the request, interacts with SQLite, and returns a response to Kong, which then returns it back through NGINX to the client.

## 4. JWT Authentication Flow

1. **User login**  
   - Client calls Kong’s `/login` route (POST) with credentials.  
   - Kong **does not** protect `/login` with JWT; request is forwarded directly to the user service.  
   - User service:
     - Verifies the username/password against SQLite.  
     - On success, issues a JWT with:
       - `iss = JWT_ISSUER`  
       - `aud = JWT_AUDIENCE`  
       - `sub = <username>`  
       - Signed with `JWT_SECRET` (HS256).

2. **Token storage**  
   - Client stores the JWT (e.g., local storage, environment variable in Postman).

3. **Accessing protected resources**  
   - Client calls the `/users` route through Kong with:  
     - `Authorization: Bearer <jwt>`.  
   - Kong JWT plugin:
     - Uses the secret for `consumer = user-service-client` (from `jwt_secrets` in `kong.yaml`) to validate the token.  
     - If signature or claims fail, responds with **401** without contacting the microservice.

4. **Microservice verification**  
   - If Kong validation passes, it forwards the request and may still pass the JWT to the backend.  
   - FastAPI backend can also decode and validate the token on its side if desired (current code primarily does its own JWT issuing and optional verify endpoint).

> **Important**: To keep signatures consistent, set the **same `JWT_SECRET` value** in:
> - Microservice env (`JWT_SECRET`)  
> - Kong Helm values (`env.jwtSecret` → `JWT_SECRET` env var in Kong)

## 5. Authentication Bypass Strategy

The assignment requires **bypassing auth** for:
- `/health`
- `/verify`

In `kong.yaml`, this is implemented by:

- Attaching the **JWT plugin only to the `/users` route** (`route: user-service-users`).  
- **No JWT plugin** is associated with `/health` and `/verify` routes.  

As a result:
- Calls to `/health` and `/verify` go through Kong without JWT validation.  
- Calls to `/users` require a valid JWT and are also subject to IP rate limiting and IP whitelisting.

## 6. Rate Limiting & IP Whitelisting

### 6.1 IP-based Rate Limiting

- Implemented via Kong’s `rate-limiting` plugin on the `/users` route.  
- Configuration (embedded in `kong.yaml` / Helm ConfigMap):
  - `minute: 10`
  - `policy: local`
  - `limit_by: ip`

**Behavior**:
- Each client IP is allowed up to **10 requests per minute** to `/users`.  
- Exceeding this limit should result in `429 Too Many Requests` responses from Kong.

### 6.2 IP Whitelisting

- Implemented via Kong’s `ip-restriction` plugin attached to the `user-service` service.
- Configuration:
  - `allow: [ ${ALLOWED_CIDR} ]` – provided via environment variable.

**Behavior**:
- Only requests with client IPs in the configured CIDR are allowed to reach the user service.  
- Requests from IPs outside the CIDR receive `403 Forbidden` from Kong.

**Configuration**:
- In `helm/kong/values.yaml`, set:
  - `env.allowedCidr` to something like your office/VPN IP range (`x.x.x.x/32` for a single IP or `x.x.x.0/24` for a subnet).

## 7. Custom Kong Lua Logic

A custom Lua plugin is provided at:
- `kong/plugins/custom.lua` (and inlined into `helm/kong/templates/configmap-plugin.yaml`).

### 7.1 Behavior

- **Header Injection**  
  - For every request to the `user-service`:
    - Adds a header (towards the upstream) with:
      - Name: `X-Platform`
      - Value: `internal-api` (configurable in plugin `config`).

- **Structured Logging**  
  - Logs a line including:
    - HTTP method  
    - Request path  
    - Client IP  
  - Useful for observing traffic patterns at the gateway level and debugging.

### 7.2 Deployment

- Plugin is mounted via a ConfigMap (`configmap-plugin.yaml`) in the Kong Helm chart:
  - Mounted at `/kong/plugins/custom.lua`.  
- `KONG_PLUGINS` is set to `bundled,custom`.  
- `KONG_LUA_PACKAGE_PATH` includes `/kong/plugins/?.lua;;`.  
- The `custom` plugin is referenced by name in `kong.yaml` under `plugins`.

## 8. DDoS Protection with NGINX Ingress + ModSecurity

### 8.1 Why NGINX + ModSecurity?

- **Open-source and self-managed**: No managed WAF service required.  
- **Kubernetes-native**: Works via the standard NGINX Ingress Controller.  
- **Flexible rules**: ModSecurity gives fine-grained control for rate limits, request inspection, and blocking logic.

### 8.2 Integration with Kong & Kubernetes

- `k8s/ingress-nginx-modsecurity.yaml`:
  - Creates `ingress-nginx` namespace.  
  - Defines a `ConfigMap` with a minimal `modsecurity.conf`:
    - `SecRuleEngine On`.  
    - IP-scoped `request_counter` with deny rule when over a threshold.  
  - Configures an **Ingress** named `kong-gateway` in namespace `platform`:
    - `kubernetes.io/ingress.class: "nginx"` – selects NGINX Ingress controller.  
    - ModSecurity enabled via `nginx.ingress.kubernetes.io/enable-modsecurity: "true"`.  
    - ModSecurity config attached via `modsecurity-snippet`.
    - Backend points to the Kong Service (ensure the name and port match your Helm release).

**Traffic path**:

`Client → NGINX Ingress (ModSecurity) → Kong Service → user-service Service`

## 9. Deployment Overview (Conceptual)

> This section lists **example steps**. Adjust image names, namespaces, and release names as needed.  
> No commands are executed by this README – they are here purely for your reference.

1. **Build & push microservice image**
   - Build FastAPI image from `microservice/Dockerfile`.  
   - Push to your registry and update image references:
     - `helm/user-service/values.yaml:image.repository`
     - `k8s/deployment.yaml:image`.

2. **Provision namespaces & base networking (Terraform)**
   - From `terraform/`, run Terraform init/plan/apply against your target cluster.  
   - This will create:
     - Namespaces: `platform`, `kong`, `ingress-nginx`.  
     - `NetworkPolicy` to restrict inbound traffic to `user-service`.

3. **Deploy NGINX Ingress Controller (if not already installed)**
   - Install NGINX Ingress (e.g., via its official Helm chart) targeting `ingress-nginx` namespace.  
   - Apply `k8s/ingress-nginx-modsecurity.yaml` so ModSecurity config and `Ingress` are created.

4. **Deploy user-service via Helm**
   - From the `helm/user-service` chart:
     - Set `image.repository` & `image.tag`.  
     - Deploy into `platform` namespace.

5. **Deploy Kong via Helm**
   - From `helm/kong`:
     - Ensure `env.userServiceHost` matches the **Kubernetes Service** name of the user-service.  
     - Set `env.jwtSecret` to match the microservice’s `JWT_SECRET`.  
     - Set `env.allowedCidr` to a secure CIDR for IP whitelisting.  
   - Deploy into the `kong` namespace.

6. **Verify connectivity**
   - Find the external IP/hostname of NGINX Ingress or the Kong Service (depending on your setup).  
   - Use that as the base URL for all testing below.

## 10. Testing Scenarios

### 10.1 Health Check (Public Endpoint)

- **Goal**: Confirm `/health` is reachable without authentication.  
- **Steps**:
  - Send `GET /health` via the external endpoint (through NGINX + Kong).  
  - Expected response: `200 OK` with body similar to `{"status": "ok"}`.

### 10.2 JWT Authentication Flow

1. **Login and get token**
   - Send `POST /login` with form data:
     - `username=admin`
     - `password=admin123` (or the configured credentials).  
   - Expected response: `200 OK` with JSON:
     - `{"access_token": "<jwt>", "token_type": "bearer"}`.

2. **Call `/users` without token**
   - Send `GET /users` without `Authorization` header.  
   - Expected response: **401 Unauthorized** from Kong due to JWT plugin.

3. **Call `/users` with token**
   - Send `GET /users` with header:
     - `Authorization: Bearer <jwt-from-login>`.  
   - Expected response: `200 OK` with a list of users (including `admin`).

4. **Verify token via public `/verify`**
   - Call `GET /verify?token=<jwt-from-login>`.  
   - Expected response: `{"valid": true, "claims": {...}}`.  
   - Note this is **public** at Kong (no auth required).

### 10.3 Rate Limiting (Kong)

- **Goal**: Demonstrate IP-based rate limiting of **10 requests per minute** on `/users`.
- **Steps**:
  1. Obtain a valid JWT (see 10.2).  
  2. From the **same client IP**, send more than 10 `GET /users` requests within one minute.  
  3. Observe:
     - First ~10 requests → `200 OK`.  
     - Subsequent requests → `429 Too Many Requests` from Kong.

### 10.4 IP Whitelisting (Kong)

- **Goal**: Show that traffic is only allowed from configured CIDR ranges.
- **Configuration**:
  - In `helm/kong/values.yaml`, set:
    - `env.allowedCidr` to a specific IP or CIDR (e.g., your dev machine IP `/32`).  
  - Redeploy/upgrade Kong with the updated values.

- **Steps**:
  1. From an **allowed IP**, call a protected endpoint (e.g., `GET /users` with valid JWT).  
     - Expected: normal behavior (200 or 429 if rate-limited).  
  2. From a **disallowed IP** (or simulate by tightening CIDR to exclude yourself and using another environment), repeat the same call.  
     - Expected: **403 Forbidden** from Kong’s `ip-restriction` plugin.

### 10.5 DDoS Protection (NGINX + ModSecurity)

- **Goal**: Show that ModSecurity blocks excessive requests before they hit Kong.
- **Steps**:
  1. From a single client IP, send a **burst of requests** (e.g., a looped script) to any endpoint (e.g., `/health`).  
  2. Observe:
     - Initially, requests pass through normally.  
     - After exceeding the configured threshold (e.g., 100+ requests in a brief window), responses start returning **429 Too Many Requests**.  
     - These responses are generated by **NGINX/ModSecurity**, not by Kong.

### 10.6 Custom Lua Plugin Behavior

- **Header injection**:
  - For a request that passes through Kong to the user-service, capture requests at the backend (e.g., by logging request headers in the microservice or via a sidecar/sniffer).  
  - Confirm that the upstream request contains header: `X-Platform: internal-api`.

- **Logging**:
  - Inspect Kong container logs.  
  - Verify presence of log lines like:
    - `[custom-plugin] method=GET path=/users client_ip=...`
